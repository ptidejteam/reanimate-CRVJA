# Project Review: Reanimate-CRVJA

## 1. Main Idea
The core concept of Reanimate-CRVJA is to bring retro Amiga AMOS BASIC games and applications to the modern web. It achieves this by functioning as a parser and transpiler that converts AMOS BASIC code into executable JavaScript code that relies on the DOM and HTML5 APIs. The project also provides an Amiga Workbench-styled web interface where users can load AMOS code, parse it, and execute it within a sandboxed environment. Additionally, it offers tools to load, extract, and visually edit AMOS sprite banks (`.abk` files).

## 2. How the Project Works (Technical Aspect)
The system is built as a Next.js web application and relies heavily on **ANTLR4** (ANother Tool for Language Recognition) for the language processing phase.


- **Lexing and Parsing**: It uses a custom grammar defined in `AMOS.g4`. ANTLR4 generates a lexer (`AMOSLexer.js`) and a parser (`AMOSParser.js`) from this grammar. The lexer tokenizes the input AMOS code, and the parser builds an *Abstract Syntax Tree* (AST).
- **AST Traversal and Translation**: `AmosToJavaScriptTranslator.js` extends `AMOSListener` (also generated by ANTLR4). As the AST is walked (using `ParseTreeWalker`), the translator listens for specific node entrances/exits (e.g., `enterScreen_open`, `enterFor_loop`, `enterPrint_something`) and appends the equivalent JavaScript code to an output string.
- **Execution Engine**: The generated JavaScript is not a direct line-by-line equivalent but includes necessary polyfills and DOM manipulation to simulate the AMOS environment. For example:
  - `Screen Open` generates a `div` acting as the screen (`#amos-screen`).
  - Graphics commands (like `Box`, `Circle`, `Turbo Draw`) generate absolute-positioned `div` elements mapped to coordinates.
  - Sound (`Play`) utilizes the Web Audio API (`AudioContext`, custom oscillators).
  - Game loops (`Do ... Loop`) are converted into `setInterval` or `setTimeout` constructs to prevent blocking the main browser thread while maintaining the game's tick rate.
  - Keyboard input (`Key State`) relies on a global event listener mapping physical keys to AMOS key codes.
- **Sandboxed Execution**: The translated JavaScript is dynamically injected into an `iframe` with appropriate `sandbox` attributes. This ensures the retro game logic runs securely within the React app (`app/page.js`) without polluting the main window's global scope.
- **Binary File Decoding**: The project contains custom logic (`AmosDecoder.js`, `bankReader.js`) to parse compiled `.amos` files and raw sprite banks (`.abk`). It extracts palette data and bitplane (planar) pixel data into a usable RGBA/hex format.

## 3. How it is Organized
The codebase is structured as a Next.js project with the following key directories and files:

- **`/` (Root)**: Contains the ANTLR grammar (`AMOS.g4`), generated ANTLR files (`AMOSLexer.js`, `AMOSParser.js`, `AMOSListener.js`), the main translator class (`AmosToJavaScriptTranslator.js`), and standard project config files (`package.json`, `next.config.mjs`, `jest.config.js`).
- **`/app`**: Contains the Next.js frontend pages and global styles. `page.js` is the monolithic main application file containing the Workbench UI, code editor with error highlighting, the sprite editor component, and the execution iframe logic.
- **`/src/tools`**: Contains helper modules:
  - `AmosDecoder.js` / `AmosCsStyle.js`: Decodes binary `.amos` files back to readable source text, loading Amiga extension libraries.
  - `bankReader/bankReader.js`: Utilities to read `.abk` sprite files.
  - `UI/workbench.js`: React components that simulate the retro Amiga Workbench window system (draggable windows, scrollbars, icons).
  - `UI/analogClock.js`: A retro analog clock widget.
- **`/public/AmosFiles`**: Example AMOS games (`Pacman.txt`, `Amos1_piano_improved.asc`, etc.) and bank files (`AmosBank_pacman.abk`).
- **`/tests`**: Comprehensive Jest test suites validating the translation of specific AMOS statements (e.g., loops, arrays, print statements, procedures).

## 4. How to Use It
1. **Installation**: Clone the repository and run `npm install` to install dependencies (Next.js, ANTLR4 runtime, React, etc.).
2. **Start the App**: Run `npm run dev` to start the Next.js development server.
3. **Using the UI**:
   - Open `localhost:3000` to see the Workbench interface.
   - Click the "CRVJA" icon to open the main code editor window.
   - You can either type AMOS code manually, click "Load .ASC File" to upload a text file, or use the "Examples" buttons to load a preset game (like Pacman).
   - If the game requires visual assets, you can load sprite banks in the designated bank slots in the UI (e.g., uploading the Pacman `.abk` file).
   - Click "Run code" to parse the code, translate it, and execute it in the "Render code area" on the right.
4. **Sprite Editing**: Click the "Sprites" icon on the desktop to open the built-in Bank Editor. Here you can load `.abk` files, modify sprite pixels on a canvas, change colors in the palette, duplicate sprites, and save the modified bank back to your local machine.
5. **Testing**: Run `npm run test` to execute the Jest test suite, ensuring the AST translation logic remains intact after making changes.

## 5. How to Create New Grammar, Rules, or Commands
Adding support for a new AMOS command involves modifying both the grammar definition and the JavaScript translator:

### Step 1: Update the Grammar (`AMOS.g4`)
1. Open `AMOS.g4`.
2. Define any new keywords as lexer tokens (e.g., `MYNEWCMD: 'MyNewCmd';`).
3. Create a parser rule for the command structure. For example, if your command takes two parameters separated by a comma:
   ```antlr
   my_new_command:
       'MyNewCmd' expression1 COMMA expression1
       ;
   ```
4. Add the new parser rule as an alternative under the main `statement` rule so the parser knows it's a valid top-level command:
   ```antlr
   statement:
       procedure
       | screen_open
       // ... existing statements ...
       | my_new_command
       ;
   ```

### Step 2: Generate the ANTLR Parser
Run the following command in the terminal to rebuild the JavaScript lexer, parser, and listener files based on your updated grammar:
```bash
java -jar .\antlr-4.13.2-complete.jar -Dlanguage=JavaScript AMOS.g4
```
*(Ensure you have Java installed and the ANTLR `.jar` file in the root directory)*

### Step 3: Implement the Translation Logic (`AmosToJavaScriptTranslator.js`)
1. Open `AmosToJavaScriptTranslator.js`.
2. Add a new `enter` (or `exit`) method matching the parser rule name exactly, but in camelCase with a capital starting letter. For `my_new_command`, you would write:
   ```javascript
   enterMy_new_command(ctx) {
       // Extract parameters from the context using the AST children
       const param1 = ctx.expression1(0)?.getText();
       const param2 = ctx.expression1(1)?.getText();

       // Generate equivalent JavaScript and append it to this.output
       this.output += `
   ${this.indent()}console.log("My new command executed with:", ${param1}, ${param2});
       `;
   }
   ```
3. Save the file.
4. Write a new Jest test in the `/tests` folder to verify your new statement generates the expected JavaScript.
5. Restart the development server, type the new command into the UI, and click "Run code" to verify that the correct JavaScript is generated and executed in the iframe.
